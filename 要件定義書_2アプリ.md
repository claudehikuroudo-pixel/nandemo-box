# 要件定義書 ─ 2アプリ開発計画

> Claude Code に渡して開発するための設計図。
> 上から順に読めば、何を・どの順で・どう作るかがわかる構成。

---

## 全体方針（2アプリ共通の思想）

両アプリは「**放り込む → AIが処理する → カード化 → 必要なら見返す／シェアする**」という同じ骨格を持つ。エンジン部分を共通化すれば開発工数が約半分になる。

| 共通原則 | 内容 |
|---|---|
| ライトユーザー特化 | 設計させない・考えさせない。Notionの逆 |
| 入口は1アクション | 放り込むだけ |
| 説明しない、体験させる | オンボーディングで最初の成功を3タップ以内 |
| 原価ゼロ思想 | 唯一の変動費はAI APIコストのみ |
| 最初から課金 | 無料で広げず、最初から有料（売れなくてもコンテンツが残る） |
| build in public | 開発過程をX・noteで発信。アプリ＝広告塔、コンテンツ＝収益 |

---

## 技術スタック（両アプリ共通）

| 層 | 技術 | 備考 |
|---|---|---|
| 画面 | React（PWA） | スマホで動く・LIMUサイトと同じ構成 |
| AI処理 | Claude API（Haiku中心） | 安い・速い・画像も読める |
| 保存（MVP） | ブラウザのローカルストレージ | サーバー不要で即動く |
| 保存（本番） | Supabase | ユーザー同期が必要になったら |
| ホスティング | Cloudflare Pages | LIMUと同じ・無料 |
| 課金 | Stripe（PWA段階） | Apple税30%を回避。手数料約3.6% |

### ⚠️ 技術検証で判明した重要な制約

- **PWAではiOSで「他アプリからのシェア受け取り」ができない**（Web Share Target APIがiOS非対応）
- スクショ選択・URL貼り付け・テキスト貼り付けは**PWAで全部動く**
- YouTube/インスタの「共有ボタン→アプリに送る」は、**App Store公開（PWABuilderまたはCapacitor）が必要**
- → MVPはPWAで作り、シェア受け取りは「App Store版の目玉」としてフェーズ2に回す

# アプリ① 気になる箱（情報おもちゃ箱）

## コンセプト

> 「気になる」を、ぜんぶ放り込む箱。
> スクショも、やることも、欲しいものも、行きたい場所も。
> 放り込めば、AIが片付けて、必要な時にそこにある。

「整理するための箱」ではなく「放り込んで楽しむ箱」。整理はAIが勝手にやる。
**見返さなくてもいい。必要な時に開けば全部ある。** Google検索と同じ思想（普段は使わない、必要な時に引く）。

## ターゲット

複数のコミュニティ・SNSから情報が洪水のように流れてきて「気になるけど忘れる」が常態化している、非エンジニア・ライトユーザーの20〜40代。

## コア体験

```
【入口】複数（全部共存）
 スクショ・写真／URL貼り付け／テキストコピペ／音声メモ
 （※シェアボタン受け取りはフェーズ2＝App Store版）
       ↓
【AI処理】自動で5分類に仕分け＋タグ付け
       ↓
【格納先】5カテゴリ
 📅予定／✅タスク／🛍モノ／📍場所／📄情報
       ↓
【出口】3つ
 ①見返す（ウォール＋タブフィルタ）
 ②向こうから来る（月1〜2回の精度の高い浮上）※フェーズ2
 ③知らなかったおすすめ（レコメンド）※フェーズ3
```

## 5分類の定義とルール

| 分類 | 中身 | 期限 | 格納先 |
|---|---|---|---|
| 📅 予定 | 日時のあるイベント・ウェビナー・講座 | あり | カレンダー＋箱 |
| ✅ タスク | やること（写真から抽出 or 手入力） | 期限あれば設定 | ToDo＋箱 |
| 🛍 モノ | 欲しいもの（時計・服・ガジェット等） | なし（熟成） | 熟成リスト |
| 📍 場所 | 行きたい店・スポット | なし | マップ／リスト |
| 📄 情報 | あとで読む記事・動画・メモ | なし | メモ |

**重要ルール：期限のあるものは種類を問わずカレンダーに集約する。** タスクも予定もセール締切も、締切があれば1つの時間軸に乗せる。「自分の未来が1画面で見える」。

## UI構造（モック確定済み）── 2層構造：上カレンダー／下ウォール

このアプリは「気になる箱」であると同時に「**ちゃんと使えるスケジューラー**」でもある。
画面を上下2層に分け、上にカレンダー（常時表示）、下に箱（ウォール）を置く。

### 上層：カレンダー（常時表示）

- **週カレンダーを常に表示**。予定がない日でも表示され、純粋なスケジューラーとして機能する
- 各日の下に**カラードット**で予定・タスク・締切の有無を示す（予定=ピンク／タスク=紫／モノ締切=黄）
- **今日**は黒く強調。日付タップで選択切り替え、選択日の予定が時刻つきでリスト表示される
- 月表示への切り替えは将来対応（週⇄月をスワイプorタップ）

### 下層：気になる箱（ウォール）

- **予定以外（タスク・モノ・場所・情報）のウォール型グリッド**（Pinterest風・カードが視覚的に並ぶ）
- タブ（すべて／タスク／モノ／場所／情報）をタップ → **該当カテゴリだけ瞬時にフィルタ**（アニメーションで再配置）
- 浮上バナーは「すべて」表示時のみ（フィルタ中は引っ込む＝集中を邪魔しない）

### 下部固定：放り込むボタン

- 「＋ 放り込む」ボタンを画面下部に常時固定。横に音声・検索のミニボタン

### この2層構造が戦略上重要な理由

**「放り込んだ後に開く理由がない（Pinterestの死に方）」問題を構造的に解決する。**
人はメモアプリは忘れるが、カレンダーは毎日開く（今日何だっけ?）。
その毎日の動線に箱を相乗りさせることで、「カレンダーを開く」習慣が箱を生かす。

```
カレンダー＝毎日必ず開く
　　＋
気になる箱＝開いたついでに放り込む・見返す
　　↓
カレンダーを開く習慣が、箱の墓場化を防ぐ
```

### 見た目の方向性（化粧は後だが方針だけ固定）

- ベースは**ライトモード（iOSウィジェット風）**。淡いグラデ壁紙＋すりガラス（フロスト）質感
- 色は5カテゴリそれぞれ鮮やかに。ただし**絵文字は本文に使わない**（安っぽくなる）。色面・グラデ・繊細な影で質感を出す
- 目指すのは「**Notionの逆**」＝説明的じゃなく感覚的、文字より画像、機能より気持ちよさ
- 参考：Apple純正アプリ・iOSウィジェット（質感）／Pinterest（眺める快感）／TikTok（直感操作）
- ※モックHTML4種を別途作成済み。最新版（nandemo_box_scheduler.html）が確定方向

## AI処理の心臓部（最重要）

放り込まれたテキスト/画像を Claude API に送り、以下のJSONを返させる。これが動けば後は表示するだけ。

```json
{
  "category": "event",
  "title": "AIゼミ・最前線",
  "date": "2026-06-17T19:00",
  "location": "オンライン",
  "url": "https://...",
  "price": null,
  "tags": ["AI", "セミナー", "SHIFT AI"],
  "confidence": 0.95
}
```

- `category` は event / task / mono / basho / joho の5択
- `confidence`（確信度）が低い時だけ、ユーザーに「これでいい?」と1回確認する

## 入口＝放り込み方（最優先実装。特に「写真」は必須）

このアプリの入口は複数あり、すべて共存させる。**中でも「写真・スクショの放り込み」が最重要の入口**。
ユーザーの実際の行動が「気になったらまずスクショを撮る」だから、ここが動かないと製品の意味がない。
（本人の利用例：LINEで届いたイベント案内、欲しい時計、行きたい店などを日常的にスクショしている）

| 入口 | 内容 | 優先度 | 実装方式 |
|---|---|---|---|
| **写真・スクショ** | カメラロールから選ぶ／撮る → Claude Vision が画像内容を読み取って分類 | **最優先・MVP必須** | `<input type="file" accept="image/*">` でbase64化 → APIに image ブロックで送信 |
| URL貼り付け | URLを貼る → 必要ならページを取得して内容を読み、分類 | MVP | テキストとして受け取り、メタ情報取得 |
| テキスト貼り付け | 案内文・メモをそのまま貼る → 分類 | MVP | textで送信 |
| 音声メモ | 「渋谷の○○行きたい」等を話す → 文字起こし → 分類 | v2 | 音声→テキスト変換を挟む |
| シェアボタン受け取り | YouTube/インスタ等の共有 → 箱に送る | v3（App Store版） | ※PWAのiOSでは不可。Capacitor等でApp Store公開時に実装 |

### 写真放り込みの実装メモ（Claude Vision）

- 画像を base64 に変換し、API メッセージの content 配列に `{type:"image", source:{type:"base64", media_type, data}}` を先頭に置く
- そのあと同じ content にテキストでプロンプト（分類指示＋JSON形式指定）を入れる
- 画像のみ（テキストなし）でも分類できるようにする＝「スクショ選ぶだけ」で完結させる
- 解析後、カードのヒーロー部分に元画像を薄く（opacity 0.5前後）残すと、見返したとき何のスクショか一目でわかる
- ⚠️ モック段階では環境により画像APIが制限される場合があるが、本番では必須機能として実装すること

## 誤分類リカバリのUX（Notion回避の核）

- カードの分類タグを**長押し**（またはタップ） → 他カテゴリの候補が出る → **1タップで移動**
- 放り込み直後の解析結果プレビューでも、その場で分類を選び直せるようにする
- AIが自信ない時（confidence低）だけ、最初に1回確認。確信ある時は黙って仕分ける
- **鉄則：直す作業が「最初から手入力するより面倒」になった瞬間に死ぬ。1〜2タップで直るを死守**

## オンボーディング（最初の3分）

```
1画面目：「試しに1つ放り込んでみましょう」＋サンプル画像が置いてある
   ↓ タップ
2画面目：AIが解析する様子を見せる → 「📅 予定として読み取りました！」
   ↓ 「勝手にやってくれた」を体感
3画面目：「あなたの番です。スクショを1枚選んでみて」→ 本物を1個入れさせる
   ↓
完了：「これだけ。あとは気になったら何でも放り込むだけ」
```

**鉄則：機能を説明しない（読まれない）。最初の成功を3タップ以内で体験させる。**

## スケジューラー機能（このアプリの第2の柱）

カレンダーを「飾り」ではなく**実用的なスケジューラー**として機能させる。
これにより「毎日開くアプリ」になり、箱の墓場化を防ぐ。以下を段階的に実装する。

| 機能 | 内容 | フェーズ |
|---|---|---|
| カレンダー常時表示 | 週表示。予定なくても常に出る | MVP |
| 予定の手動追加 | 放り込みだけでなく、タップで普通に予定作成 | MVP |
| 選択日の予定リスト | 日付タップで、その日の予定を時刻つき表示 | MVP |
| 月表示切り替え | 週⇄月をスワイプ/タップ | v2 |
| **Googleカレンダー双方向同期** | 既存予定が全部出る／このアプリで足した予定もGoogleに入る | v2（重要） |
| 通知・リマインド | 予定の前に知らせる | v2 |
| 繰り返し予定 | 毎週の道場・定例など | v2 |

### ⚠️ Googleカレンダー同期は必須級

スケジューラーとして本気で使ってもらうなら、Googleカレンダー双方向同期は避けて通れない。
「このアプリだけ予定が孤立する」状態だと、メインのスケジューラーとして使われず、結局開かれなくなる。
MVPではローカルで動かしてよいが、**v2で必ず同期を入れる**前提で設計する。



## 開発フェーズ

| フェーズ | 作るもの | build in public ネタ |
|---|---|---|
| **MVP**（今週末） | ①週カレンダー常時表示＋予定の手動追加　②放り込む（スクショ/URL/テキスト）→ AI仕分け → ウォール＋タブ表示。ローカル保存 | 「スケジューラー＋気になる箱が1画面に」 |
| **v2** | Googleカレンダー双方向同期／月表示／通知・リマインド／浮上（月1〜2回の精度の高い通知）。Supabase導入 | 「Googleカレンダーと繋がった」「忘れてた気になるに救われた」 |
| **v3** | App Store公開（シェアボタン対応）＋レコメンド＋熟成リスト | 「PWAからApp Storeアプリに進化」 |

## 浮上（v2の核）の設計

- 頻度ではなく**精度**で勝つ。月1〜2回、「自分でも忘れてたドンピシャ」だけ届ける
- 例：「3ヶ月前に気になってた渋谷の店、今週末予定が空いてます」
- 押し付けない。Google検索の関連キーワードのように、開いた時にそっと出る

# アプリ② 怒られダイエット（毒舌コーチアプリ）

> ※過去セッションで設計済み。ここでは要点を再掲し、アプリ①と共通化する前提で整理。

## コンセプト

食事写真を撮る → AIがカロリー・PFCを推定 → **毒舌オニコーチが怒る/褒める** → それを「キャラカード」1枚画像に変換 → ワンタップでSNS共有。

例：深夜の柿ピーに「**有罪・約300kcal・歩45分で帳消し**」。

## 科学的前提

セルフモニタリング効果。「消費カロリー以下に抑えているか記録するだけで痩せる」という、本人の80→73kg減量の実体験が中心的な約束。運動は必須にしない。

## ターゲット

ダイエットしたいが続かない人。自虐・笑いで継続したい層。本人（減量経験者）が当事者として地で語れる。

## コア体験

```
食事写真を撮る
   ↓
AIがカロリー・PFC推定（±20-30%の誤差は許容、厳密さより継続）
   ↓
毒舌オニコーチが判定（キャラの人格は選択可能）
   ↓
「有罪」カード1枚画像に自動変換（スタンプ風デザイン）
   ↓
ワンタップでSNS共有 ← これがコアループに組み込まれている
```

## アプリ①との共通エンジン

両アプリは骨格が同じ。共通化して工数を半減させる。

| 共通部分 | 気になる箱 | 怒られダイエット |
|---|---|---|
| 放り込む | スクショ・URL・テキスト | 食事写真 |
| AI処理 | 5分類に仕分け | カロリー・PFC推定 |
| カード化 | カテゴリカード | 有罪/無罪カード |
| 出口 | 見返す・浮上 | SNS共有 |

→ **「放り込む→AI処理→カード生成」のエンジンを共通モジュールとして1回作る。** 各アプリは入力と出力の味付けを変えるだけ。

## MVP機能（過去セッションで確定済み）

AIキャラ判定／ワンタップ共有／バーコードスキャン／栄養トラッキング（アフィリリンク付き）／手動ワークアウト記録（動画共有）など13項目。

## フェーズ2以降（やらないことの明確化）

自動レップカウント・フレンドグループ・before/after写真編集・キャラチャット等は後回し。**自動レップカウント等の作り込みはコアメッセージをぼやけさせるので最初は避ける。**

## North Star（最重要KPI）

**週あたりのシェアされたカード枚数。** これが伸びれば全部回る。

## SNS的な強み（散財ストッパーより上）

| | 散財ストッパー | 怒られダイエット |
|---|---|---|
| 当事者性 | 薄い（演技） | ◎ 自分が減量経験者 |
| ネタ供給 | 捻り出す→尽きる | 自動で湧く（毎日の食事＝ネタ） |
| 拡散 | 開発過程だけ | アプリ自体が拡散装置 |

ネタ元が3重：①開発過程（build in public）②自分のダイエット記録（当事者）③アプリが生むシェアカード。

# 価格設計（両アプリ共通の考え方）

| プラン | 価格 | 狙い |
|---|---|---|
| 無料お試し | 7日間 | 「放り込む快感」を体験させる |
| 月額 | ¥480〜780 | ※下記の判断による |
| 年額 | 月額×10ヶ月分 | 年払い誘導でLTV最大化 |
| 買い切り | 出さない | AIコストが続くため非推奨 |

### 価格の判断軸（未確定・本人判断待ち）

- **480円**：入りやすさ重視。DL数・口コミが伸びやすい＝build in publicのネタになりやすい
- **780円**：単価重視。安物感を回避。値下げ余地を残せる
- 競合実勢（2026年）：Raindrop約3ドル/月、Matter約5ドル/月、Readwise約10ドル/月

### 収益シミュレーション

```
480円 × 300人  = 月14.4万（年173万）
480円 × 1,000人 = 月48万（年576万）
780円 × 300人  = 月23.4万（年280万）
780円 × 1,000人 = 月78万（年936万）
```

**戦略：1アプリで月20万を「作って走らせとくだけ」で複数並べる。** 1個コケても他が走る。ポートフォリオで積み上げるのが個人開発の勝ち方。
